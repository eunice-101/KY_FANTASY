# 나만의 글쓰기 워크플로우 (My Writing Workflow)

단편 콘텐츠를 위한 5단계 구조적 글쓰기 프로세스.

## 스타일 로딩

이 스킬은 `my-style.md` 파일에서 사용자의 글쓰기 스타일을 읽어온다.
- `my-style.md`가 존재하면: 해당 스타일을 모든 Step에 적용한다
- `my-style.md`가 없거나 비어있으면: 아래 기본 스타일을 적용한다

**사용자의 `my-style.md`를 반드시 먼저 확인하고, 그 안의 설정값을 모든 단계에서 반영한다.**

---

## 전체 흐름

```
Step 1: 목적 정의 → Step 2: 구조 설계 → Step 3: 초안 작성 → Step 4: 리라이팅 → Step 5: 최종 출력
```

사용자의 상태에 따라 적절한 단계부터 시작한다.
- 주제만 있음 → Step 1부터
- 구조/아웃라인 있음 → Step 3부터
- 초안 있음 → Step 4부터

### Step 간 전환 규칙

각 Step 완료 후, 다음 Step으로 넘어가기 전에 사용자 확인을 받는다.

| 전환 | 확인 방식 |
|------|----------|
| Step 1 → 2 | 목적·독자·채널·유형을 요약 제시 → 확인 후 진행 |
| Step 2 → 3 | 아웃라인 제시 → 확인 후 진행 |
| Step 3 → 4 | 초안 제시 → 확인 후 진행 |
| Step 4 → 5 | 수정본 제시 → 확인 후 진행 |

"끝까지 진행해줘" 또는 "한 번에 해줘" → 확인 없이 연속 진행.

## 기본 글쓰기 원칙 (my-style.md가 없을 때 적용)

### 문체 기준
- 표준어 사용, 설명형·해설형 서술 중심
- 종결어미 기본값: '~합니다'체
- 문장은 18~25단어 내외 기본
- **한 문장에는 하나의 정보, 하나의 판단만 담는다**
- 문장은 능동형을 기본으로 한다
- 문단은 3~5문장을 기본으로 한다
- 줄바꿈은 문단 단위로만 적용

### 서술 전략
- 글의 첫 문장: **독자의 문제 상황 또는 핵심 질문**으로 시작. 기능·현상 나열로 시작하지 않음
- 설명 순서: **개념 → 판단 기준 → 실제 적용 방식**
- 소제목은 기능명이 아닌 **행동·판단 중심 문장**으로 작성
- 핵심 기준·판단 포인트는 **볼드체**로 강조
- 마무리는 핵심 요약 + 독자의 다음 행동(CTA) 제시

### 금지 요소
- 정보 나열식 구성 (리스트만 던지고 해설 없는 글)
- 결과물에 대한 감상형 표현
- 과도한 반복 설명
- 근거 없는 과장·미사여구

### 금지어 규칙 (AI 글쓰기 흔적 제거)
- **① 과장형 형용사 금지:** 혁신적인, 획기적인, 창의적인, 강력한, 완벽한 → 수치·사례로 대체
- **② 모호한 부사 금지:** 매우, 굉장히, 쉽게, 빠르게 → 시간·단계·수치로 대체
- **③ 추측·완곡 표현 금지:** ~할 수 있습니다, ~일 수 있습니다 → 단정형 서술
- **④ 불필요한 연결어 금지:** 전반적으로 볼 때, 이러한 측면에서 → 근거→주장 직접 연결
- **⑤ 추상 명사 중심 문장 지양:** 중요성, 필요성, 가능성 → 동사 중심 전환
- **⑥ AI 흔적 조합 금지:** 부사+형용사+가능성+추상명사 패턴 금지
- **⑦ AI 문장 구조 패턴 금지:** ~라고 할 수 있습니다, 비로소 ~됩니다, 한층 더 → 삭제 후 단축

### 분량 통제 규칙
- **"다듬어줘"** = 표현만 정리, 구조 유지
- **"보강해줘"** = 지정 부분만 추가
- **"추가해줘"** = 지정 위치에 지정 분량만
- 120% 초과 시 사유 확인

## Step 1: 목적 정의 (Purpose Definition)

### 수집 항목

| 항목 | 질문 | 예시 |
|------|------|------|
| **목적** | 이 글로 무엇을 달성하려는가? | 설득, 교육, 정보 공유, 브랜딩, 기록 |
| **독자** | 누가 읽는가? 현재 수준은? | 입문자, 실무자, 의사결정자, 일반 대중 |
| **채널** | 어디에 게시하는가? | 블로그, 뉴스레터, 링크드인, 사내 보고 |
| **콘텐츠 유형** | 어떤 형식인가? | 아래 유형 중 선택 |

### 콘텐츠 유형 분류

| 유형 | 특징 | 기본 분량 |
|------|------|----------|
| 블로그/아티클 | SEO 고려, 스캔 가능한 구조 | 1,500~3,000자 |
| 뉴스레터 | 개인 톤, 정기 발행, CTA 포함 | 800~1,500자 |
| 칼럼/기고문 | 관점 중심, 논증 구조 | 1,500~2,500자 |
| 분석 기사 | 주제 심층 분석, 판단 기준 제시 | 4,000~6,000자 |
| 보고서/제안서 | 구조적, 데이터 기반 | 2,000~5,000자 |
| 보도자료 | 역피라미드, 객관적 톤 | 800~1,200자 |
| SNS 롱폼 | 경험 중심, 스토리텔링 | 1,000~2,000자 |
| 강의 스크립트 | 구어체, 시간 단위 구성 | 분당 약 300자 |

### 산출물

4가지 항목 요약 → 사용자 확인 → Step 2 진행.

## Step 2: 구조 설계 (Structure Design)

### 유형별 구조 골격

**블로그/아티클:**
```
도입: 독자의 문제 또는 질문 제기
본론 1: 핵심 개념과 판단 기준
본론 2: 실제 적용 방식 + 예시
본론 3: 자주 발생하는 실수 또는 심화
마무리: 핵심 정리 + CTA
```

**뉴스레터:**
```
인사/도입: 이번 호의 핵심 메시지
메인 콘텐츠: 1~2개 주제 (간결하게)
부가 정보: 추천 자료, 링크, 팁
마무리: CTA + 다음 호 예고
```

**칼럼/기고문:**
```
도입: 논점 제기 (사건, 현상, 질문)
논증 1: 근거 + 해석
논증 2: 반론 검토 또는 보강
결론: 저자의 판단 + 제언
```

**분석 기사:**
```
도입: 현상 또는 문제의 재정의
본론 1: 핵심 개념과 판단 기준
본론 2: 구조/메커니즘 설명
본론 3: 실전 사례
마무리: 관점 재정렬
```

**보고서/제안서:**
```
요약(Executive Summary)
배경/현황
분석/핵심 내용
제안/권고사항
기대 효과
부록
```

### 산출물

아웃라인 제시 → 사용자 확인 → Step 3 진행.

---

## Step 3: 초안 작성 (First Draft)

### 초안 작성 규칙

- Step 2 구조를 따라 작성한다
- `my-style.md`의 스타일을 적용한다 (없으면 기본 원칙 적용)
- 문단은 4~6줄 이내 유지
- 완성도보다 흐름 우선 — 막히는 부분은 `[TODO: ...]` 마커 삽입
- 채널에 맞는 톤 적용

### 톤 가이드

| 채널/유형 | 기본 톤 | 특징 |
|----------|---------|------|
| 블로그 | 친근한 전문가 | 쉬운 설명, 독자에게 말 거는 느낌 |
| 뉴스레터 | 신뢰하는 동료 | 개인적 어조, 솔직함 |
| 칼럼 | 분석적 관찰자 | 근거 기반, 절제된 표현 |
| 분석 기사 | 구조적 해석자 | 현상 재정의, 깊이 있는 사례 |
| 보고서 | 객관적 전문가 | 데이터 중심, 판단 근거 명시 |
| SNS 롱폼 | 경험 나누는 실무자 | 스토리텔링, 교훈 도출 |

**`my-style.md`에 채널별 톤이 설정되어 있으면 그것을 우선 적용한다.**

### 산출물

초안은 대화창 텍스트로 제공 (요청 시 파일 저장). 사용자 확인 후 Step 4 진행.

---

## Step 4: 리라이팅 (Rewriting)

4가지 관점으로 점검하고 수정한다.

### 진행 방식

1. **점검 리포트 제시** → 2. **사용자 확인** → 3. **수정본 작성** → 4. **변경 요약**

### 점검 1: 구조 점검

| # | 점검 항목 |
|---|----------|
| 1 | 도입부가 독자의 문제/질문으로 시작하는가 |
| 2 | 전체 흐름이 논리적 순서를 따르는가 |
| 3 | 소제목이 행동·판단 중심인가 |
| 4 | 마무리가 핵심 정리 + CTA로 끝나는가 |

### 점검 2: 문장 점검

| # | 점검 항목 |
|---|----------|
| 1 | 문장이 18~25단어 범위 내인가 |
| 2 | 종결어미가 통일되어 있는가 |
| 3 | 단정적 서술을 사용하는가 |
| 4 | 과장/AI 흔적 표현이 없는가 |
| 5 | 핵심 포인트가 볼드 강조되어 있는가 |

### 점검 3: 스타일 점검

**`my-style.md`에 설정된 스타일과 일치하는지 점검한다.**

| # | 점검 항목 |
|---|----------|
| 1 | 사용자가 설정한 어조와 일치하는가 |
| 2 | 사용자가 설정한 "좋아하는 문장 스타일"이 반영되었는가 |
| 3 | 사용자가 설정한 "싫어하는 스타일"이 나타나지 않는가 |
| 4 | 사용자의 "나만의 무기"가 활용되었는가 |
| 5 | 사용자의 "금기 사항"을 위반하지 않았는가 |

### 점검 4: 채널 최적화

| # | 점검 항목 |
|---|----------|
| 1 | 분량이 채널 기준에 맞는가 (±20%) |
| 2 | 톤이 채널에 적합한가 |
| 3 | 제목/소제목이 매력적이고 명확한가 |

### 변경 요약 포맷

```
**주요 변경 사항:**
- 구조: (변경 내용)
- 문장: (변경 내용)
- 스타일: (변경 내용)
- 채널: (변경 내용)
```

## Step 5: 최종 출력 (Final Output)

### 최종 점검 항목

| # | 점검 항목 | 기준 |
|---|----------|------|
| 1 | 제목 확정 | 후보 3~5개, 핵심 키워드 포함, 15자 이내 권장 |
| 2 | 소제목 정리 | 스캔 가능, 행동·판단 중심 |
| 3 | 맞춤법·문법 교정 | 한국어 맞춤법, 띄어쓰기, 외래어 표기법 |
| 4 | 메타 정보 (블로그) | 메타 디스크립션, 태그 |
| 5 | [TODO] 마커 해결 | 모든 마커 해결 또는 제거 |
| 6 | 분량 최종 확인 | 유형별 기본 분량 범위(±20%) |
| 7 | 스타일 최종 체크 | `my-style.md` 설정과 최종 대조 |

### 산출물

| 파일 | 설명 |
|------|------|
| `final.docx` 또는 `final.md` | 최종 완성본 |

출력 포맷은 사용자 지정에 따름. 기본값은 `.docx`.
문서 생성 전 반드시 `/mnt/skills/public/docx/SKILL.md`를 읽고 따른다.
